Continue implementation from the scaffold and make the backend operational for the provided multi-family foundation while extending it beyond read-only behavior.

Use the provided stack and finish wiring so the local workflow succeeds end-to-end:

- `npm install --no-audit --no-fund`
- `npm run prisma:generate`
- `npm run prisma:migrate`
- `npm run seed`
- `npm run build`
- `npm start`

Seeded auth expectations:

- `parent@example.com` / `family123` must log in and retain at least two family memberships.
- `caregiver@example.com` / `caregiver123` must log in and retain a narrower membership set used for tenancy isolation checks.
- `child@example.com` / `child123` must log in and retain a single-family read-only membership.
- Login must return a bearer token in one of these keys: `accessToken`, `token`, `bearerToken`, or `jwt`.
- Protected family routes must accept `Authorization: Bearer <token>` using that token value.
- In the shared family membership used by parent/caregiver/child:
  - caregiver permissions include `canManageTasks: false`, `canPost: true`, `canViewLocation: true`.
  - child permissions include `canManageTasks: false`, `canPost: false`, `canViewLocation: false`.

Required read routes:

- `GET /healthz`
- `POST /api/v1/auth/login`
- `GET /api/v1/families`
- `GET /api/v1/families/:familyId/summary`
- `GET /api/v1/families/:familyId/tasks`
- `GET /api/v1/families/:familyId/events`
- `GET /api/v1/families/:familyId/feed`
- `GET /api/v1/families/:familyId/lists`
- `GET /api/v1/families/:familyId/reminders`
- `GET /api/v1/families/:familyId/locations`

Read response envelope contract:

- `GET /api/v1/families` returns `{ "families": [...] }`.
- `GET /api/v1/families/:familyId/summary` returns `{ "family": { ... } }`.
- `GET /api/v1/families/:familyId/tasks` returns `{ "tasks": [...] }`.
- `GET /api/v1/families/:familyId/events` returns `{ "events": [...] }`.
- `GET /api/v1/families/:familyId/feed` returns `{ "posts": [...] }`.
- `GET /api/v1/families/:familyId/lists` returns `{ "lists": [...] }`.
- `GET /api/v1/families/:familyId/reminders` returns `{ "reminders": [...] }`.
- `GET /api/v1/families/:familyId/locations` returns `{ "locations": [...] }`.

Required write routes (must persist and be visible in subsequent reads):

- `POST /api/v1/families/:familyId/tasks`
- `PATCH /api/v1/families/:familyId/tasks/:taskId`
- `POST /api/v1/families/:familyId/events`
- `POST /api/v1/families/:familyId/feed`
- `POST /api/v1/families/:familyId/feed/:postId/comments`
- `POST /api/v1/families/:familyId/feed/:postId/likes`
- `POST /api/v1/families/:familyId/lists`
- `POST /api/v1/families/:familyId/lists/:listId/items`
- `POST /api/v1/families/:familyId/reminders`

Write request payload contract (minimum accepted fields):

- `POST /tasks` accepts `{ "title": string, "cadence": "ONE_TIME" }` and may also accept `visibility`.
- `PATCH /tasks/:taskId` accepts `{ "status": "DONE" }`.
- `POST /events` accepts `{ "title": string, "startsAt": ISO8601, "endsAt": ISO8601 }` and may also accept `visibility`.
- `POST /feed` accepts `{ "body": string }`.
- `POST /feed/:postId/comments` accepts `{ "body": string }`.
- `POST /feed/:postId/likes` accepts an empty or omitted JSON body.
- `POST /lists` accepts `{ "title": string }` and may also accept `visibility`.
- `POST /lists/:listId/items` accepts `{ "label": string }`.
- `POST /reminders` accepts `{ "title": string, "scheduleText": string }`.

Write response envelope contract:

- Create/update routes return JSON containing the created/updated entity as either a named key
  (`task`, `event`, `post`, `comment`, `list`, `item`, `reminder`) or a direct object with `id`.
- The updated task response should surface status `DONE` when patched to done.

Data and tenancy requirements:

- Family summary includes aggregate counts for `tasks`, `events`, `reminders`, `lists`, and `posts`.
- Feed includes social engagement metadata (`commentCount`, `likeCount`).
- Task, event, feed, and location records include `familyId` and it matches the requested `:familyId`.
- Task/event/list records surface `visibility`.
- Seeded reads for tasks/events/feed are non-empty for at least one accessible family.
- In the shared family, seeded tasks/events/lists include both `PRIVATE` and `SHARED` records.
- In the shared family, seeded location check-ins are non-empty.
- For tasks, events, and lists in the shared family:
  - `OWNER`/`ADMIN` can read both `PRIVATE` and `SHARED` records.
  - `CAREGIVER`/`CHILD` reads must exclude `PRIVATE` records.
- Permission gates for family-scoped writes:
  - `canManageTasks` controls task create/update, event create, list create/item create, and reminder create.
  - `canPost` controls feed post create, feed comment create, and feed like create.
- Authorization source of truth:
  - Membership and permission authorization must be enforced from current database state, not only from JWT claims.
  - If `canPost` is revoked in DB after login, the same token must lose post/comment/like access (`403`).
  - If family membership is removed in DB after login, the same token must lose family access (`403`).
- Location access gate:
  - `GET /locations` requires both membership `canViewLocation=true` and `family.locationSharingEnabled=true`.
  - Parent location reads on their non-shared secondary family should return `403` (seeded location-sharing-disabled path).
- For a forbidden family, these read routes must return `403`:
  `/summary`, `/tasks`, `/events`, `/feed`, `/lists`, `/reminders`, `/locations`.
- For a forbidden family, these create routes must return `403`:
  `POST /tasks`, `POST /events`, `POST /feed`, `POST /lists`, `POST /reminders`.

Write-path validation expectations:

- Each required write route should return JSON and expose the created/updated entity id (`task`, `event`, `post`, `comment`, `list`, `item`, `reminder`) either as a named object field or direct response object.
- The task status update route must persist the requested status change and remain visible in the subsequent task listing.
- The created event must appear in a subsequent events read for the same family.
- The created post must appear in a subsequent feed read for the same family.
- After creating a comment and a like for the created post, that post should report `commentCount >= 1` and `likeCount >= 1`.
- The created list and created list item must appear in a subsequent lists read for the same family.
- The created reminder must appear in a subsequent reminders read for the same family.
- After the write sequence, family summary aggregate counts for `tasks`, `events`, `reminders`, `lists`, and `posts` must each increase by at least 1 from baseline.
- Caregiver write behavior in the shared family:
  - task create and task status update (`PATCH /tasks/:taskId`) return `403`.
  - event create, list create, list-item create (`POST /lists/:listId/items`), and reminder create return `403`.
  - feed post/comment/like writes succeed.
- Child write behavior in the shared family:
  - task and feed writes return `403`.
- Location behavior:
  - parent and caregiver can read shared-family locations.
  - child receives `403` on shared-family locations.
